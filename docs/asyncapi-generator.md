# AsyncAPI Generator

`protoc-gen-connect2asyncapi` generates one AsyncAPI v3.1 JSON document for the
JSON WebSocket routes generated by `protoc-gen-connect2ws`.

The generator documents the actual WebSocket protocol shape:

- unary RPCs are omitted because `connect2ws` does not generate unary
  WebSocket routes;
- server-streaming RPCs receive one JSON request frame and send many JSON
  response frames;
- client-streaming RPCs receive many JSON request frames, use an empty text
  frame as the end-of-stream marker, and send one JSON response frame;
- bidirectional-streaming RPCs receive many JSON request frames and send many
  JSON response frames;
- server-streaming methods with path/query REST bindings are skipped to match
  `connect2ws`.

## Buf Usage

```yaml
plugins:
  - local: protoc-gen-connect2asyncapi
    out: src/generated/asyncapi
    strategy: all
    opt:
      - config=connect2asyncapi.yaml
```

## Plugin Options

Options are comma-separated `name=value` pairs.

| Option | Default | Purpose |
| --- | --- | --- |
| `output` | `asyncapi.json` | Generated output file name. |
| `config` | none | YAML config file for metadata, servers, and security schemes. |
| `server_url` | none | Convenience URL used as the default AsyncAPI server. |
| `default_content_type` | `application/json` | AsyncAPI `defaultContentType` and message content type. |

## Config File

The config file reuses the shared API document metadata shape where practical.
`servers` may be written in OpenAPI-style URL form; the generator converts it
to AsyncAPI `host`, `protocol`, and `pathname`.

```yaml
info:
  title: WebSocket Streaming API
  version: 0.1.0
  description: WebSocket streaming connect2axum example API.

servers:
  - url: ws://127.0.0.1:8002/v1
    description: Local WebSocket server

securitySchemes:
  BearerAuth:
    type: http
    scheme: bearer
    bearerFormat: JWT

security:
  - BearerAuth: []
```

Channels are keyed by the generated WebSocket route path, such as
`/hello/chat/ws`. Each channel has `request` and `response` messages that
reference reusable `components.messages`, which in turn reference
`components.schemas` built from protobuf message descriptors.
